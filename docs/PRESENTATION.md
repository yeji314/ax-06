# 🏠 부동산 추천 Agent

> LangGraph 기반 부동산 상담사형 AI Agent

---

## 1. 소개

### 저희 조는 **부동산 매매 조회 Agent**를 만들었습니다.

복잡한 부동산 검색을 **마치 상담사와 대화하듯**
자연어로 조건을 말하면 최적의 매물을 찾아주는 Agent.

```
"왕십리 매매 20억 이하 아파트 추천해줘"
         ↓
    Agent가 알아서
    - 조건을 분석하고
    - 매물을 찾고
    - 검증한 뒤
    - 추천 멘트를 생성
```

- **팀원**: AX 6조
- **기술 스택**: LangGraph · LangChain · OpenAI GPT-4o-mini · **Tavily 웹 검색** · Python 3.9+
- **목표**: 부동산 초심자도 쉽게 접근할 수 있는 대화형 매물 추천

---

## 2. 개발 의도

### 💭 부동산을 사겠다고 결심하는 순간

처음엔 **가격**과 **위치** 두 가지만 생각하지만…

```
  가격 → "헉, 비싸다"
     ↓
  세세한 조건 → "세대수? 주차? 계단식/복도식? 연식?"
     ↓
  공부해야 할 게 너무 많음 😵
     ↓
  포기하거나 막막해짐
```

### 🎯 그래서 만들었습니다

> **"부동산 상담사처럼, 질문하면 대답해주는 Agent"**

- 부동산 시장 진입 장벽을 낮춘다
- 복잡한 필터 대신 **자연어 대화**로 매물을 탐색
- **불충분한 질문 → 되묻기**로 초심자도 자연스럽게 조건을 구체화
- **검증 단계**로 엉뚱한 추천을 방지

## 3-A. 구현 요소

### 🧱 State 정의 (`AgentState`)

```python
class AgentState(TypedDict):
    user_input: str                    # 사용자 원본 입력
    condition: UserCondition           # 파싱된 조건
    is_valid: bool                     # 필수조건 충족 여부
    error_message: Optional[str]
    search_results: List[dict]         # Agent가 생성한 매물
    filtered_results: List[dict]       # 필터링·검증 통과 매물
    recommendations: str               # 최종 추천 텍스트
    retry_count: int                   # 입력 명확화 재시도 횟수
    verify_retry_count: int            # 검증 실패 재검색 횟수
    messages: List[Any]
```

**UserCondition** — 3대 필수 + 10개 선택 조건

| 구분 | 필드 |
|------|------|
| **필수** | 지역 · 거래유형(매매/전세/월세) · 가격 |
| 선택 | 방 유형 · 면적 · 세대수 · 주차 · 계단식/복도식 |
| 선택 | 역까지 분 · 방/욕실 개수 · 층/방향 · 연식 |

## 3-B. State Diagram

```
              ┌──────────────────────┐
              │        [START]       │
              └──────────┬───────────┘
                         ▼
              ┌──────────────────────┐
              │    parse_condition   │  LLM: 자연어 → JSON 조건
              └──────────┬───────────┘
                         ▼
              ┌──────────────────────┐
              │      validate        │  지역·유형·가격 3가지 필수 체크
              └──────────┬───────────┘
                         │
            ┌────────────┴────────────┐
       조건 부족                  조건 충족
            ▼                        ▼
   ┌────────────────┐       ┌─────────────────────────────┐
   │    clarify     │◀─루프─│          search             │
   │ (되묻기+예시)   │       │  ① Tavily 웹 검색 (실시간)   │
   └────────────────┘       │  ② LLM이 검색결과+조건으로   │
                            │     매물 생성 → 필터·점수화   │
                            └──────────────┬──────────────┘
                                           ▼
                            ┌────────────────┐
                            │     verify     │  가격/유형/지역 재검증
                            └────────┬───────┘
                                     │
                      ┌──────────────┴─────────────┐
                 통과 0건                       통과 ≥ 1건
                      │ (재시도 < 2)                 │
                      └─▶ search 재실행              ▼
                                            ┌────────────────┐
                                            │   recommend    │  LLM: 추천 멘트 생성
                                            └────────┬───────┘
                                                     ▼
                                                  [END]
```

## 3-C. 노드별 역할

| # | 노드 | 역할 |
|---|------|------|
| 1 | **parse_condition** | 자연어를 JSON 조건(17개 필드)으로 LLM 파싱 |
| 2 | **validate** | 필수 3조건(지역·유형·가격) 존재 여부 검증 |
| 3 | **clarify** | 부족한 조건을 **예시와 함께 되묻기** (사과 X, 재질문 O) |
| 4 | **search** | **Tavily 웹 검색** → LLM이 결과 근거로 매물 6개 생성 → 필터·점수화 |
| 5 | **verify** | **가격·유형·지역 필수조건 재검증** → 0건 시 `search` 재실행 |
| 6 | **recommend** | 장단점·추천 이유 포함한 자연어 추천 텍스트 생성 |

### 핵심 설계 철학

> "Agent는 단순 실행자가 아니라, **스스로 판단하고 돌아갈 줄 아는 존재**"

- **질문 평가 → 단계 회귀**: clarify 루프 (조건 부족 시 parse로 복귀)
- **결과 평가 → 단계 회귀**: verify 루프 (검증 실패 시 search로 복귀)

## 3-D. Tool 설계

### 🌐 `search_web` (Tavily 실시간 웹 검색)
- 조건을 기반으로 **네이버 부동산·호갱노노·실거래가 등**을 실시간 검색
- `include_domains=[land.naver.com, hogangnono.com, rtms.molit.go.kr ...]` 으로 **신뢰 도메인 우선**
- API 키 없으면 빈 결과 반환 → 전체 흐름 안 끊김 (graceful fallback)

### 🤖 `llm_generate_properties` (LLM 매물 생성)
- 웹 검색 결과를 **LLM 프롬프트 context로 주입**
- LLM이 "검색된 실제 단지명/시세를 우선 참조해 매물 JSON 생성"
- 구조화된 17개 필드(세대수·주차·구조·방향·연식…) 형태로 출력

### 🧮 `filter_and_score`
- 전체 조건으로 2차 필터링 및 점수화
- 가격(+30) · 면적(+20) · 역세권(+15) · 주차(+5) · 옵션 보너스

### 🔄 데이터 흐름
```
조건 → Tavily 웹 검색 ──▶ 실시간 시세·단지 정보
              │
              ▼
          LLM에 주입 ──▶ 검색 근거 + LLM 지식을 결합해 매물 6개 생성
              │
              ▼
      filter_and_score ──▶ 점수 정렬 + 상위 5건
              │
              ▼
          verify ──▶ 가격/유형/지역 필수조건 재검증
```

> **핵심**: 실데이터 근거 + LLM의 자연어 이해를 조합해 **hallucination을 크게 줄임**

## 3-E. 예시 — 정상 시나리오

**입력**
```
🏡 어떤 매물을 찾으시나요? 마포구 월세 보증금 3000 월 80 이하 투룸
```

**파싱된 조건**
```
지역: 마포구 | 거래유형: 월세 | 최대보증금: 3000 | 최대월세: 80 | 방종류: 투룸
```

**웹 검색 로그** (Tavily)
```
🌐 웹 검색 5건 수집 → LLM 프롬프트 주입
  - 마포구 투룸 월세 시세 — 네이버 부동산
  - 공덕동 투룸 매물 리스트 — 호갱노노
  - 마포구 실거래가 분기 동향 — 국토부 RTMS
  ...
```

**검증 로그**
```
🛡️ 필수조건 검증: 가격 / 유형 / 지역
  ✅ 통과 [L001] 마포구 공덕동 투룸
     - 가격: ✅ 보증금 3000/월 80 ≤ 조건
     - 유형: ✅ 거래유형 월세 일치, 방 유형 투룸 일치
     - 지역: ✅ '마포구' 매칭
📌 6건 중 4건 통과
```

**추천 결과**
```
🏆 1순위: 마포구 공덕동 투룸
  - 5호선 공덕역 도보 5분 · 남향 45m²
  - 보증금 3,000 / 월 80 · 예산 완벽 충족
```

## 3-F. 예시 — 불충분한 질문

**입력**
```
🏡 어떤 매물을 찾으시나요? 왕십리 아파트 추천해줘
```

**validate 결과**
```
필수 조건 누락: 거래유형, 가격
```

**Agent의 재질문** (clarify)
```
🤔 추가 정보가 필요해요:
구체적으로 어떤 거래 방식과 예산을 생각하고 계실까요?
- 거래 유형: 월세 / 전세 / 매매 중 어떤 것인가요?
- 예산: 예) '매매 20억 이하', '전세 5억 이하' 등
```

**사용자 보충 입력 → parse 재진입 → validate 통과 → search 진행**

## 3-G. 예시 — 검증 재검색

**입력**
```
🏡 어떤 매물을 찾으시나요? 강남구 월세 보증금 500 월 30 이하
```

**첫 번째 검증**
```
❌ 탈락 [L001] 강남 원룸 — 가격: 월세 70 > 30
❌ 탈락 [L002] 강남 오피스텔 — 가격: 월세 85 > 30
📌 6건 중 0건 통과
♻️ 통과 매물 0건 → 재검색 시도 1회차
```

**Agent가 `search`로 되돌아가 재생성 → 다시 verify**

> "결과가 사용자 기대와 다르면 스스로 다시 시도한다"

## 4-A. 회고 — 배운 점

### 💡 핵심 인사이트

> **"부동산은 검색이 아니라 상담이다"**
> — 우리는 이를 Agent로 구현하고자 했다

| 처음 생각 | 실제로 깨달은 것 |
|-----------|-----------------|
| 조건만 주면 필터링하면 끝 | 질문 자체가 불완전한 경우가 대부분 |
| 1-shot LLM 호출로 충분 | State Machine으로 **단계별 평가/회귀**가 핵심 |
| Mock 데이터로 데모 | LLM 단독 생성은 hallucination 위험 → **웹 검색 근거 주입**이 현실적 |
| 네이버 부동산 공식 API 필요 | 공식 API 없음 → **Tavily 도메인 필터링**으로 대체 가능 |

### 🏗️ LangGraph의 강점 체감
- 조건부 엣지(`add_conditional_edges`)로 **상황별 분기** 자연스럽게
- 각 노드는 작고 명확 — 디버깅·확장이 쉬움
- `State` 중심 설계로 **모든 단계의 맥락이 투명**

## 4-B. 회고 — 추가 개발 방향

### 🚀 확장 계획

**① State 확장**
```
단순 (유형 · 위치 · 가격)
         ↓
다차원 조건 구조
  └─ 목적(실거주/투자) · 선호 환경 · 교통 · 학군 · 커뮤니티 ...
```

**② 사용자 의도 파악 고도화**
- "실거주 vs 투자" 같은 **목적 기반 분기** 추가
- 예: 투자 목적 → 수익률·전세가율·거래량 중심 추천
- 예: 실거주 목적 → 학군·편의시설·출퇴근 중심 추천

**③ RAG / 추천 시스템 결합**
```
단순 조회형 Agent  →  개인화 추천형 Agent
  - 과거 대화 이력 기반 선호도 학습
  - 유사 매물 RAG 검색
  - 사용자 페르소나 기반 랭킹
```

**④ 실데이터 연동 심화**
- **현재**: Tavily 웹 검색으로 네이버 부동산·호갱노노·실거래가 정보를 LLM 근거로 주입 ✅
- **다음**: 공공데이터포털 실거래가 API 직접 연동 → 검증 단계에서 실거래 가격대와 대조
- **최종**: 직방·다방 공식 제휴 API → 실시간 매물 리스트 직접 조회

## 4-C. 회고 — 아쉬웠던 점

### ⚠️ 한계

**평가 단계의 모호성 판단**
- 질문의 모호함을 **완벽하게 판단하지 못하는 경우** 존재
- 예: "싸게 좋은 집" — 주관적 조건은 파싱 실패
- 예: "학군 좋은 데" — 정량화되지 않은 조건

**LLM 생성 매물의 hallucination** (일부 해소)
- Tavily 웹 검색으로 **실제 단지명·시세 정보를 근거로 주입** → 완화
- 다만 개별 매물 수준의 실시간 시세까지는 아직 반영 한계
- 공식 실거래가 API 연동이 남은 과제

**재검색 루프의 한계**
- 검증 0건 시 최대 2회 재시도 후 포기
- 근본 원인이 "비현실적 조건"일 때는 루프만으로 해결 불가
- → **조건 완화 제안**으로 우회 중

### 🛠️ 개선 방향
- 모호한 질문 감지 모델 별도 분리
- 검증 실패 원인별 차등 처리 (가격 위반 → 완화 제안 / 지역 위반 → 재생성)
- 실데이터 기반 검증으로 hallucination 제거
