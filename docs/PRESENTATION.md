# 🏠 부동산 추천 Agent

> LangGraph 기반 부동산 상담사형 AI Agent · **국토부 실거래가 실데이터 + 자연어 깊이 추론**

---

## 1. 소개

### 저희 조는 **부동산 매매·전월세 조회 Agent**를 만들었습니다.

복잡한 부동산 검색을 **마치 상담사와 대화하듯**, 자연어로 조건을 말하면
**진짜 실거래 데이터에서** 매물을 찾아주는 Agent.

```
"학군 좋고 한강 근처 회사가 시청역인데
 1시간 이내 출근 가능한 매매 15억 이하 아파트"
              ↓
   Agent가 알아서:
   - 학군→강남·서초·양천 후보 추출
   - 한강 인접 동만(35개 동) strict 필터
   - 시청역에서 60분 이내 통근 매트릭스 적용
   - 국토부 실거래 API 호출
   - 가격·유형·지역 필수조건 재검증
   - 추천 멘트 생성
```

- **팀원**: AX 6조 · **트랙**: ① Workflow
- **기술 스택**:
  - LangGraph(StateGraph + MemorySaver) · Pydantic structured output
  - OpenAI GPT-4o-mini · **국토교통부 실거래가 API** · Tavily · LangSmith
  - FastAPI + Streamlit + Rich CLI 3가지 인터페이스

## 2. 개발 의도

### 💭 부동산 진입 장벽

```
처음엔 가격·위치만 생각 → 알아갈수록 늘어나는 조건들
   "세대수? 주차? 구조? 연식?"
   "도보? 통근 시간?"
   "학군? 한강뷰? 분위기?"
   "외국인 거주 비율?"
              ↓
   포기하거나 막막해짐 😵
```

### 🎯 그래서 만들었습니다

> **"부동산 상담사처럼, 자연어로 말하면 진짜 매물을 찾아주는 Agent"**

- ✅ 실거래 **공공 데이터** 기반 (LLM hallucination 차단)
- ✅ 모호한 자연어 표현(`학군 좋은`·`한강 근처`·`회사가 시청역`)도 자동 해석
- ✅ **불충분한 질문**은 되묻고, **결과 0건**일 땐 진단·재시도
- ✅ 멀티턴 대화로 조건 누적 ("20억으로 다시", "강남구도 추가")

## 3-A. 구현 요소 — State 정의

### `UserCondition` — 21개 매물·환경 조건

| 분류 | 필드 |
|------|------|
| **필수 4** | region · deal_type · property_type · price |
| 매물 스펙 | min_area · min_households · parking_required · building_structure · min_rooms · min_bathrooms · preferred_floor · top_floor_only · direction · max_building_age |
| 환경·생활 | max_subway_minutes · exclude_high_foreign_density · hangang_view_only · commute_from · max_commute_minutes |

### `UserLifestyle` — 4개 생활권 키워드

```python
class UserLifestyle:
    activities:   List[str]   # ["런닝","자전거","등산","수영","헬스"]
    atmosphere:   Optional[str]  # "조용한"/"활발한"/"카페거리"/...
    amenities:    List[str]   # ["공원","한강","카페","헬스장"...]
    raw_keywords: Optional[str]
```

### `AgentState` — 그래프 전역 상태

```python
class AgentState:
    user_input · condition · lifestyle · is_valid
    clarify_question                  # API가 프론트로 반환 후 그래프 종료
    search_results · filtered_results
    filter_stats                      # 단계별 탈락 통계 (진단용)
    recommendations
    retry_count · verify_retry_count · relaxed
    messages
```

## 3-B. State Diagram

```
            ┌─────────────────────────┐
            │       [START]           │
            └────────────┬────────────┘
                         ▼
            ┌─────────────────────────┐
            │   parse_condition       │  Pydantic + 16개 정규식 보강
            │   (자연어 → JSON 조건)   │
            └────────────┬────────────┘
                         ▼
            ┌─────────────────────────┐
            │      validate           │  지역·유형·방종류·가격 필수
            └────────────┬────────────┘
                         │
              ┌──────────┴──────────┐
        조건 부족             조건 충족
              ▼                   ▼
       ┌─────────────┐   ┌─────────────────────────┐
       │  clarify    │   │   search_and_filter     │
       │  질문 생성  │   │ ① 광역→대표구·인접구·동 │
       │  + 그래프   │   │ ② MOLIT 실거래가 API    │
       │   종료      │   │ ③ 20+ 하드 필터          │
       └─────┬───────┘   │ ④ 점수화 → 상위 5건     │
             │           └────────────┬────────────┘
             │  다음 invoke            ▼
             │  (MemorySaver           ┌──────────┐
             │   자동 머지)            │  verify  │ 가격/유형/지역
             │                         └────┬─────┘
             │                              │
             │                  ┌───────────┴───────────┐
             │             통과 0건                 통과 ≥ 1
             │                  │ (재시도<2)             │
             │                  └─▶ search 재실행        ▼
             │                                    ┌──────────────┐
             │                                    │  recommend   │ + Tavily 동네 정보
             │                                    └──────┬───────┘
             │                                           ▼
             └────  사용자 답변  → parse 재진입       [END]
```

**두 개의 자기복원 루프**

> "Agent는 단순 실행자가 아니라, **스스로 판단하고 돌아갈 줄 아는 존재**"

| 루프 | 트리거 | 동작 |
|------|--------|------|
| **clarify** | 질문 부족 (validate 실패) | 부족 조건 LLM 재질문 → state에 저장 후 그래프 종료 → 다음 invoke 시 MemorySaver가 머지 |
| **verify 재검색** | 결과 0건 (verify 실패) | 인접 구 +1 확장 + 소프트 조건 완화 (`relaxed=True`) → 최대 2회 재시도 |

## 3-C. 노드별 역할

| # | 노드 | 핵심 동작 |
|---|------|----------|
| 1 | **parse_condition** | Pydantic structured output + 억 단위 자동 교정 + 16개 정규식 보강 + 멀티턴 머지 |
| 2 | **validate** | 4대 필수 조건(지역·유형·방종류·가격) 체크 / `retry_count ≥ 2` 안전 통과 |
| 3 | **clarify** | 부족 항목 + 예시로 LLM 질문 생성 → state에 저장 후 종료 (멀티턴) |
| 4 | **search_and_filter** | 광역→대표구·인접구·역 도보권 동 확장 → MOLIT API → 20+ 필터 + 점수 |
| 5 | **verify** | 가격·유형·지역 재검증 / 0건 시 search로 자기복원 (최대 2회) |
| 6 | **recommend** | 매물 + Tavily 동네 정보로 자연어 추천. 0건 시 막대그래프 진단 |

## 3-D. 자연어 깊이 추론 — 16개 정규식 보강

LLM의 단점(환각·누락·추론오류)을 정규식 안전망 16개로 보강:

```
사용자: "D2호선 근처 한강 학군 좋은
       회사 시청역 1시간 이내 매매 15억 아파트 탑층"
                    ↓
   ① 정제: "D2호선" → "2호선"
   ② 호선: 2호선 → region 보강 (마포·성동·강남)
   ③ 한강 근접: hangang_view_only=True → 35개 한강 인접 동만
   ④ 학군 키워드: lifestyle.amenities 추가
   ⑤ region 백필: lifestyle → 강남·서초·양천 후보 추가
   ⑥ 통근 거점: "회사 시청역" → commute_from
   ⑦ 통근 시간: "1시간 이내" → 60분
   ⑧ 억 단위 교정: 15,000 → 150,000만원
   ⑨ 탑층 패턴: top_floor_only=True
   ⑩ 추론 차단: deal_type/property_type 명시 안 됐으면 거부
```

| 보강 항목 | 효과 예시 |
|-----------|----------|
| 입력 정제 | `D2호선` → `2호선` (IME 오타) |
| 호선·역·동 추출 | `옥수역` → 옥수동·금호동·응봉동 도보권 |
| 광역 region 처리 | `서울` 단독 → `학군` 키워드로 강남·서초·양천 자동 |
| 한강 근접 패턴 | `한강 근처/뷰/주변` → 한강 인접 35개 동 strict 필터 |
| 외국인 부정 패턴 | `중국인 많지 않은` → 외국인 밀집 7개 동 제외 |
| 통근 거점·시간 | `회사가 시청역, 1시간 이내` → 5개 거점 × 25구 매트릭스로 검증 |
| LLM 환각 차단 | `15억`만 말해도 LLM이 매매로 추론 → 거부 후 clarify 진입 |

## 4-A. 회고 — 배운 점

### 💡 핵심 인사이트

> **"부동산은 검색이 아니라 상담이다"**
> "그리고 LLM은 데이터 소스가 아니라 **인터페이스**다."

| 처음 생각 | 실제로 깨달은 것 |
|-----------|-----------------|
| 조건 입력만 받으면 끝 | 질문 자체가 불완전한 경우가 압도적 |
| LLM이 조건 잘 뽑아줄 것 | 16개 정규식 안전망 없으면 환각·누락이 심각 |
| 1-shot 호출로 충분 | State Machine + **2개의 자기복원 루프**가 핵심 |
| Mock JSON 데모 | LLM 단독 생성 → 공공 실거래가 API 전환이 정답 |
| 네이버 부동산 공식 API 필요 | 공식 API 없음 → **국토부 RTMS**로 충분히 대체 |
| LLM이 모든 걸 한다 | LLM은 파싱·재질문·추천 멘트만, **나머지는 룩업·필터·휴리스틱** |
| 한 번에 다 말해야 함 | **멀티턴**이 자연스러움 (MemorySaver) |

### 🏗️ LangGraph의 강점 체감

- 조건부 엣지(`add_conditional_edges`)로 **상황별 분기** 자연스럽게
- 각 노드는 작고 명확 → 디버깅·확장 용이
- `State` 중심 설계로 모든 단계의 맥락이 투명
- **MemorySaver checkpointer**로 멀티턴 자동 처리
- **LangSmith @traceable**로 LLM 호출 trace 가시화

## 4-B. 추가 개발 방향

### 🚀 데이터 소스 다각화

```
✅ 국토부 실거래가 API (구현 완료, 메인 소스)
✅ Tavily 동네 정보 검색 (추천 멘트용)
🔜 KOSIS 외국인 통계 API → 동 단위 인구통계 정밀화
🔜 학교알리미 API → 학군 정량 점수 (수능·진학률)
🔜 카카오 로컬 + ODsay → 실시간 통근 계산
🔜 직방·다방 제휴 → 호가 데이터 (실거래는 이력만)
```

### 🚀 Agent 고도화

- **목적 기반 분기**: 실거주 vs 투자 → 추천 우선순위 다르게
- **개인화 추천**: 과거 대화 이력 기반 선호도 학습 (RAG)
- **의도 감지 모델 분리**: 모호한 질문 vs 구체적 질문 구분
- **이미지 분석**: 매물 사진을 LLM이 보고 "한강뷰", "도배 좋음" 자동 태깅

## 4-C. 회고 — 아쉬웠던 점

### ⚠️ 데이터 자체 한계

**MOLIT 실거래가 API의 빈 구석**
- 지하철역 거리 정보 없음 → "역까지 N분" 강성 매칭 불가
- 총층수 정보 없음 → "탑층" 강성 매칭 불가
- **현재 매물이 아닌 과거 거래 이력** → 실시간 호가는 직방 등 별도 필요
- 빌라 분류 거침 → 도시형생활주택·연립·다세대 구분 정확도 낮음

### ⚠️ 휴리스틱 정확도

- **통근 시간**: 5개 거점 × 25구 정적 매트릭스 (±10분 오차)
- **빌라 정규식**: `OO빌` suffix 패턴으로 자동 분류 → 새 브랜드 누락 가능
- **외국인 밀집 동**: 7개 동 하드코딩 → KOSIS 연동 필요

### ⚠️ LLM 한계

- "싸게 좋은 집" 같은 **주관적 조건**은 정량화 불가 → clarify 거쳐 분리해야 함
- LLM이 가끔 `"중구 아파트"` → `property_type='중아파트'` 같은 합성 오류 (화이트리스트로 방어 중)

### 🛠️ 개선 방향

- 모호한 질문 감지 모델 별도 분리 (clarify 정밀도 ↑)
- 검증 실패 원인별 차등 처리 (가격 위반 → 완화 / 지역 위반 → 인접 구)
- 실데이터 기반 검증으로 hallucination 완전 제거
- 빌라 분류 정교화 (KAIST 부동산 brand 데이터셋 등)
